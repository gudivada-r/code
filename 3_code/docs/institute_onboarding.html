EdNex Institute Onboarding Protocol

Standard Operating Procedure for provisioning new Multi-Tenant University Systems
Architecture Philosophy: EdNex utilizes a highly scalable Multi-Tenant Architecture. Entire University Systems (like TAMUS) inhabit a single, isolated Supabase instance. Individual sub-colleges within a system are natively segregated using the university_id relational enforcement coupled with Row Level Security (RLS).
Step 01

System-Level Provisioning

Infrastructure

Provision a new Supabase Enterprise project dedicated to the University System. Establish VPC peering if the institution demands secure, private-network data ingestion from their on-premise SIS.

Step 02

Multi-Tenant Schema Configuration

Database Security

Execute the EdNex Modular Initialization scripts across the blank database. Ensure that the university_id column is properly indexed across all modules (Mod-00 -> Mod-05).

Apply Supabase Row Level Security (RLS) policies to guarantee that an API request with an Auth Token belonging to "College A" can never access records from "College B".

-- Example RLS Enforcement CREATE POLICY "Isolate by University ID" ON public.mod01_student_profiles FOR SELECT USING ( university_id = (auth.jwt() ->> 'assigned_university_id')::uuid );
Step 03

Establish ETL Data Pipelines

Data Engineering

Integrate the newly created Supabase instance with the institution's existing legacy systems using tools like Airbyte or Fivetran.

Map the standard EdNex modular tables to the institution's localized systems:

  • Mod-01 (SIS): Banner, Workday Student, Colleague.
  • Mod-02 (Finance): PeopleSoft, TouchNet.
Step 04

Unstructured Knowledge Synchronization

Vector Ops

Trigger the EdNex Document Parser. Point the ingestion webhooks to the institution's Learning Management Systems (Canvas, Blackboard) and public HTTP directories.

The parser will automatically chunk, embed, and store syllabus PDFs, campus maps, and student handbook policies straight into the pgvector database module tagged with the specific university_id.

Step 05

Configure The Intelligence Layer (LLM)

Generative AI

Map the institution's Gemini API Keys to the backend environment variables. The API Gateway will now automatically intercept incoming student prompts.

The Gateway's Context Assembler will silently append the student's live GPA (from Mod-01), current financial standing (from Mod-02), and the student's most recent interaction history memory (from ConvDB) into the prompt string before passing it to the language model.

Step 06

Single Sign-On (SSO) Integration

Authentication

Configure SAML 2.0 or Microsoft Entra ID (Azure AD) connections inside the Supabase Auth Dashboard. Map the incoming external provider's university domain to automatically assign the user's correct university_id upon their first login.

Step 07

Integration Validation & Handover

QA

Log into the aumtech.ai web portal using a System Administrator account. Access the Admin Panel and navigate to the EdNex Module Status tab.

Verify that all 15 operational data modules return a green "Operational" health status indicating that data from all sub-colleges has been successfully routed and indexed securely into the multi-tenant architecture.
